JetPack JoyRide

A playable game built in hardware, where your voice is the controller - a real-time FFT of mic input drives the player.

Summary

  • Designed and implemented a complete FPGA-based game, using real-time DSP and a Fast Fourier Transform of microphone input to build a player-control system driven by vocal pitch.
  • Architected it on hardware-design principles: the AXI bus for colour rendering, lookup tables for sprite generation, and SPI for keyboard networking.

The problem

On an FPGA there is no runtime, no scheduler and no frame you can quietly drop. Everything is concurrent, everything has to meet timing, and the display controller is going to ask for the next pixel whether or not anything upstream is ready for it.

Voice control sharpens that. Pitch detection is a frequency-domain question, and the frequency domain is not free: an FFT needs a window of samples before it can say anything at all. So the game has a deadline the signal processing does not naturally respect, and the design has to reconcile the two in fabric rather than by scheduling around it.

How it works

Microphone samples are windowed and transformed in hardware; the dominant frequency bin is read off as pitch, and pitch becomes the player’s vertical control. Sing higher, climb. There is no threshold on amplitude standing in for control - the control signal is the pitch itself.

The video path is deliberately cheap. Sprites live in lookup tables rather than being computed, so the pixel pipeline is a memory read on a fixed schedule, and colour data moves over the AXI bus. Keyboard input comes in over SPI. Every one of those choices is the same choice: put the work where it costs logic instead of where it costs time.

What I’d change

Pitch is a fragile control channel. It is sensitive to room noise, it behaves differently for different voices, and it has no neutral position - silence is not a pitch, so the system needs a policy for "no input" that a button gives you for free.

The real design tension is the FFT window. A longer window gives finer frequency resolution and therefore steadier control, and costs exactly that much latency between the player’s voice and the player’s avatar. Somewhere in that trade there is a window length that feels best, and I picked it by ear rather than by measuring what players actually preferred.
