A from-scratch recreation of Granola, the AI meeting notepad: a local-first Electron app that listens to a meeting, transcribes it, and fuses the user’s own sparse notes with the machine transcript into clean structured notes.

The constraint that defined the project: exactly one runtime dependency. The WAV codec, the audio chunker, the transcript stitcher, the OAuth client, the Markdown renderer, and the SSE parser are all hand-written — because the point was to understand the primitives rather than assemble them.

The core idea

Note enhancement is not summarization, and that distinction is the whole product. The user’s sparse shorthand notes are the backbone and the priority signal — they encode what the human actually cared about. The transcript is the fact source used to expand that shorthand into complete points.

It runs under strict grounding rules: never invent names, numbers, dates or commitments; everything the user noted must appear and must not be buried under model-discovered points; match their voice; silently correct obvious ASR mishears.

Engineering decisions worth defending

  • No native modules. System-audio loopback comes from Electron’s display-media handler rather than a virtual audio cable or kernel driver — the single decision that keeps the dependency count at one.
  • Deterministic where possible, probabilistic only where necessary. Seam de-duplication is arithmetic on word-level timestamps, not a model call. The meeting scheduler is pure bookkeeping. The LLM is used only where language understanding is the actual task.
  • Different models for different jobs, chosen by measurement. One model gave a 12-second time-to-first-token — unusable for streaming UI. A faster one gave 2.4s. So the app runs the strong model for one-shot enhancement and the fast one for streaming chat.
  • Immutable source of truth. The pristine transcript is never overwritten, so glossary corrections always recompute from source and can never stack or compound.
  • Deliberately no RAG. An hour of speech is roughly 13k tokens and fits in context; retrieval would be complexity with no accuracy benefit.
  • Testability as a design input. The two hardest components — the rolling audio buffer and the meeting timer — were built pure, with injectable clocks and timers, specifically so they could be tested exactly rather than approximately.

A negative result, and the pivot

The intended fix for mangled domain terms was the ASR’s prompt parameter, which biases the decoder toward supplied spellings. It doesn’t work through this gateway: tested across four models on a real clip, output was byte-identical with and without the prompt.

The pivot was to correct the output instead of steering the input — a temperature-0 pass with a deliberately narrow mandate: fix only mishearings of terms implied by the user’s glossary, preserve every other word exactly, never rephrase or reorder, and leave anything ambiguous alone.

Hand-rolled OAuth

Calendar integration runs the full desktop Authorization Code + PKCE + loopback flow with only Node core APIs — no Google client library. A throwaway server binds to a kernel-assigned port, consent opens in the system browser rather than an embedded webview, and a single-settle guard means error, success, and timeout can never double-resolve.

Tokens are encrypted at rest through the OS crypto backend. The app never records automatically: when a scheduled meeting starts it surfaces a prompt, and recording begins only on an explicit click.

Designed, not yet built

Speaker attribution is specified but unshipped, and the reasoning is the interesting part. The app captures device audio — but the conferencing app has already mixed every remote participant into one output stream before the OS hands it over. The separation happened upstream; no source separation recovers it.

Acoustic speaker clustering was rejected on privacy grounds (it’s voice fingerprinting, against the local-first posture). Joining as a bot was rejected because appearing in the participant list is precisely what this class of product exists to avoid. The designed approach instead reads the active-speaker highlight the conferencing app is already displaying on screen.
