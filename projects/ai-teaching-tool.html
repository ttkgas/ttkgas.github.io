AI Teaching Tool

An AI tutor built outward from user research - a RAG pipeline that remembers what a student already asked.

Summary

  • Led comprehensive user research, interviewing students, parents and teachers to identify the real pain points in the educational journey - a lack of personalized support, and fear of asking questions in class.
  • Architected a full-stack proof-of-concept: a React frontend for interactivity, a vector-database backend, and gRPC for low-latency real-time communication.
  • Engineered a Retrieval-Augmented Generation pipeline that chunks and embeds conversational data into a ChromaDB vector store, so the model gives context-aware responses that reference previous interactions.

The problem

The finding that shaped the whole build came out of the interviews rather than out of a design session: students often do not ask, because asking is socially expensive. That reframes the product. The bottleneck is not the availability of explanations - explanations are abundant - it is the cost of admitting you need one.

But a tutor that answers and then forgets is just a chatbot with a subject. What makes tutoring work is that the tutor remembers you were confused about this same thing last week. So the technical problem became continuity: how does a model know what this particular student has already been told?

How it works

The conversation is the corpus. Each exchange is chunked and embedded into a ChromaDB vector store, and retrieval pulls the relevant slices of the student’s own history back into context before the model answers. That is the whole idea: personalization is a retrieval problem, not a prompting problem, and it gets better the longer a student uses it instead of decaying as the context window fills.

The frontend is React and the transport is gRPC, streaming so answers appear as they are produced rather than arriving as a block. For a tutor that matters more than it looks - a visible pause reads as the system not understanding you, which is exactly the feeling the product exists to remove.

What I’d change

Retrieving over your own conversation has a failure mode I underweighted: it makes the model’s past mistakes durable. A wrong explanation given on Tuesday is a retrievable document by Thursday, and the system will happily cite itself. Any serious version needs a correction and eviction path so a bad turn can be removed from the store rather than reinforced by it.

The other gap is grounding. Retrieval over conversation answers "what have we discussed" well and "is this actually correct" not at all. The curriculum needs to be a second, authoritative retrieval source that outranks the chat history whenever the two disagree.
