Write Yourself a Git

Git rebuilt from the bytes up - object store, refs, and the staging index, by hand.

A working Git implementation in pure Python - no third-party dependencies, just the standard library and a careful reading of how .git/ stores history on disk. I rebuilt the object store, the reference system, and the binary staging index by hand to stop treating version control as a black box.

With the internals understood, I’m extending past the tutorial into real Myers diff and three-way merge.

Summary

  • Rebuilt Git’s object store from scratch - SHA-1 content addressing, zlib compression, and all four object types.
  • Parsed the binary staging index by hand to power add, status, and commit.
  • Shipped fifteen plumbing and porcelain commands - init, hash-object, cat-file, ls-tree, ls-files, rev-parse, show-ref, add, rm, status, commit, log, checkout, tag, check-ignore - with zero third-party dependencies.
  • Now extending past the tutorial into Myers diff and three-way merge via lowest common ancestor.

What I learned

Git is a content-addressable filesystem with a version-control interface bolted on top, and almost every confusing thing about it stops being confusing once you have written the storage layer. A blob is bytes addressed by the hash of those bytes. A tree is a list of names pointing at hashes. A commit is a tiny file naming one tree and its parents. That is the entire data model, and it fits in a paragraph.

Two things genuinely changed how I reason about the tool. First, the index is a real third thing. It is not a staging metaphor, it is a binary file on disk with its own format, and status is literally a three-way comparison between HEAD, that file, and the working tree. Once you have parsed it by hand, "staged" and "unstaged" stop being vocabulary and start being locations.

Second, branches are cheap because they are nothing. A branch is a forty-byte file containing a hash. Deleting one deletes a pointer, not history - which is why the recovery advice that sounds like folk wisdom is actually just a consequence of the storage design.

Going beyond the tutorial

The tutorial ends where the algorithms begin, which is the interesting part. Two commands are in progress and both are real computer science rather than format parsing.

diff is tree-to-tree and blob-to-blob comparison producing genuine unified-diff output, with a Myers diff computing the line-level changes. The object model makes half of this free - comparing two trees is mostly hash equality, and identical subtrees can be skipped wholesale without reading a byte of content. The cost concentrates in the blobs that actually differ.

merge is the harder one. Three-way merge means first finding the merge base - the lowest common ancestor of two commits in the history graph - and only then reconciling base, ours and theirs into a merged tree with conflict detection where the histories genuinely disagree. The lesson so far is that the merge base is the whole problem: reconciliation is mechanical once you know what the two sides diverged from. Next up is branch management as a first-class command, fast-forward versus true merge-commit handling, and conflict markers compatible with standard tooling.
