Language to locomotion · Edge LLM · Jan – Aug 2025

Observing AI Agent

A local LLM on a Jetson turning a natural-language instruction into motor commands, with no cloud in the loop.

Summary

  • Developing a control system for an autonomous robot by running a local large language model on an NVIDIA Jetson, so the agent parses natural-language commands and makes its own decisions on board.
  • Implemented a working proof-of-concept that translates LLM text output directly into physical motor actions over MCP, demonstrating a complete language-to-locomotion pipeline.

The problem

A language model emits text. A robot consumes velocities. Nobody hands you the layer in between, and it is not a parsing problem - it is a question of what the model is even allowed to say. Free-form text that a script pattern-matches into motion is a system where a rephrasing is a crash.

Running the model on the robot rather than in a datacenter is the other half. Once something has mass and moves, network latency and connectivity stop being performance characteristics and start being safety characteristics. A robot that stops thinking when the wifi drops is not autonomous, it is remote-controlled with extra steps.

How it works

Ollama hosts the model directly on the Jetson. The motion primitives the robot can actually perform are exposed to it as MCP tools, so the model does not describe what it wants in prose - it calls something. That turns the text-to-motion boundary into a typed interface: the model can only express commands the robot can execute, and anything outside that vocabulary fails at the boundary instead of halfway down the drive chain. ROS carries the resulting command to the motors.

Choosing tool-calling over prose parsing is the decision I would defend hardest. It is the difference between a demo and a control system.

What I’d change

The observing half is the unfinished half, and it is the harder one. Right now the pipeline is essentially open loop: language goes in, motion comes out, and the model never learns whether the motion accomplished anything. A real agent needs the perception result fed back into its context so the next decision is conditioned on what actually happened, and that loop is where all the interesting failure modes live.

The Jetson is also a genuine constraint rather than a detail. A local model trades reasoning quality for latency and independence, and I have not measured where that trade stops being worth it - which is exactly the measurement that would tell you whether this design generalises past a proof-of-concept.
