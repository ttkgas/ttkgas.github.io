A model-agnostic multi-agent harness built with spec-driven development: a frontier model orchestrates a crew of cheaper models over the cloud to improve token-usage efficiency, exposed to any client through an MCP server. Written from scratch in Python with no agent framework underneath it - no LangChain, no LangGraph, no CrewAI - because framework lock-in was the one thing the design could not afford.

Summary

  • Architected a model-agnostic multi-agent harness using a frontier model to orchestrate a crew of models over the cloud, improving token-usage efficiency, exposed via an MCP server.
  • Implemented verification gates against hallucination and security risk - a path-scope allowlist and exit-code verification backed by trusted integration tests, plus a command-verification classifier to stop malicious code.
  • Created profile distillation from the work a model has already done, informing a deterministic router for future delegation.

The problem

The premise is economic. Frontier-model tokens are the scarce resource, so the obvious move is to push parallelizable work down to cheaper models. The obvious move fails for two reasons, and both of them are about trust.

The crew grades its own homework. The first version decided a delegation had succeeded because the model emitted STATUS: COMPLETED. That single line meant every downstream statistic - the ledger, the capability profiles, the routing decisions built on both - was ultimately sourced from a model’s opinion of itself. Reputation data inflated by optimistic self-reporting is worse than no data, because it is confidently wrong.

Reviewing the work costs what the work saved. The default failure mode of lead-plus-crew systems is that the lead re-reads and re-derives every returned diff, which discards the entire payoff. If verifying delegated work costs frontier tokens proportional to the work itself, you have bought nothing.

And underneath both: a delegated model drives a bash tool. Its commands must never execute on the host as me.

How it works

Nine layers, each depending only downward, with nothing model-specific anywhere in the core. A model plugs in as a declarative YAML manifest parsed into a typed spec - slug, context window, token budget, reasoning effort, capability grants, pricing - so adding a model is adding a file and swapping one is changing a string. That claim was built as a falsification test: a second provider model was added with zero core code changes, and if it had required any, the design was wrong.

The handoff is a contract, not a wish. A task carries the instruction, minimal interface context, explicit acceptance criteria, a path allowlist, and a verification command. The criteria and the scope are rendered into the prompt because complying with them is the goal - but the verify command is deliberately withheld, because showing a model the exact check invites it to satisfy the check instead of the requirement.

Two deterministic, zero-token gates replaced self-grading. A scope allowlist compiles the declared path globs into regexes and flags any changed file outside them, which catches ordinary scope creep exactly as well as it catches malicious edits. Then verification runs an arbitrary command in the sandbox whose exit code is ground truth. Any flag downgrades the result to quarantined, which the ledger records as a non-success. This is also the answer to cold start: the very first delegation is trustworthy because the check is objective, not because the model has earned a reputation yet.

Execution is fail-closed. Each command runs in a throwaway Docker container with only the workspace bind-mounted, so the .env is not merely unreadable but absent; no network, non-root, all capabilities dropped, no-new-privileges, read-only root filesystem, and cgroup caps on memory, CPU and PIDs. If Docker is unreachable the workspace refuses to construct, so delegation can never silently degrade into running untrusted model-authored code on the host. The isolation is proven rather than asserted: a self-test runs six adversarial probes inside a live container and reports pass or fail on each.

Outcomes compound. Every delegation appends a row to an append-only ledger keyed by model and task type, aggregated into success rates and token and dollar costs, with recommendations gated on a minimum sample count so the router says "insufficient data" rather than inventing confidence from one lucky run. Distillation then folds those outcomes into each model’s capability profile - a deterministic zero-token qualification stage first, and only if something clears it, exactly one LLM call that rewrites the profile rather than appending to it, so the injected document stays bounded forever while the raw log stays the unbounded archive.

All of it is exposed over stdio JSON-RPC as an MCP server, which is what removed the friction that had been quietly suppressing delegation: when delegating means writing a CLI invocation, the lead under-reaches for the crew.

What I’d change

The spend controls were reactive. Four layers of budget enforcement - a per-delegation cap checked before the next paid call, full cost accounting, a swarm aggregate cap, and a fail-closed account-credit floor - exist because telemetry surfaced a single delegation burning 329,582 tokens and still failing. Every one of those was foreseeable. Cost should have been a first-class quantity from the first paid call, not a subsystem retrofitted after a bad ledger row.

The parallel budget cap is honest but leaky. Crossing the aggregate cap stops the queue; it does not kill jobs already in flight, so worst-case overshoot is bounded by in-flight jobs times the per-job cap. That is documented rather than fixed, which is the right call for now and the wrong one eventually.

Where it breaks down: the whole approach rests on work being cheaply verifiable by an exit code. That covers a great deal of engineering and covers almost none of design, naming, API shape, or anything where the acceptance criterion is taste. For those the gate has nothing to say, so the lead has to read the diff after all - and the system correctly makes no promises there. Extending verification past exit codes without falling back into model self-assessment is the open problem.
